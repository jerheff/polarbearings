## Verdict

**Overall health: B+ / Healthy. No verified correctness or compatibility bugs; rigorous numerical testing, honest performance docs, SHA-pinned least-privilege CI, and verified-correct multi-version compat handling are genuine strengths. The deductions are entirely pre-1.0 polish: pervasive, partly self-contradictory documentation drift (the most embarrassing for a first release since README ships to PyPI), an API keyword-only inconsistency with a real positional-weight footgun, a structural test-gap the coverage gate cannot see, and a missing trusted-publishing release path. All are fixable before tagging 0.1.0; most are quick wins.**

polarbearings is in strong shape for a pre-1.0 library. The audit surfaced NO correctness, compat, or runtime bugs that survive verification — the numerical core matches scikit-learn where it claims to, the compat fixes (d2_pinball 1.24.0 panic, feature-detected fused bootstrap path) are verified-correct, and the performance documentation is honest and reproducible (it even retracts an earlier overclaim). The real issues cluster in three buckets: (1) documentation drift — every human-facing and agent-facing metric count is stale and self-contradictory (README says 37 metrics / "two exceptions"; actual __all__ is 63 with 6 unweighted; TESTING.md lists 5 nonexistent test files and a nonexistent ci.yml workflow); (2) pre-1.0 API-consistency polish — the entire pre-existing code family (classification + regression + ranking) leaves weight/pos_label/threshold positional while every newly-added family forces keyword-only, producing a real footgun where f1_score('y','p','w') silently mis-binds weight to threshold and crashes cryptically; (3) test/CI hardening — the universal null-policy contract is untested for ~15 regression metrics (the 100% line-coverage gate structurally cannot catch a forgotten guarded()), DCG/NDCG tie handling is order-dependent and diverges from its documented sklearn-parity claim under ties, and there is no automated/trusted-publishing release workflow. None of these block usefulness today; several should be fixed before tagging 0.1.0 because they ship in the PyPI description or affect the supply chain. Two findings were refuted outright (MAPE divergence is already documented; the ECE/MCE 512-element warning does not reproduce at the default n_bins=10), and several were down-scoped during verification.

### 2. Positional weight/pos_label/threshold inconsistency: classification (and all pre-existing families) leave them positional while every new family forces keyword-only; f1_score('y','p','w') silently mis-binds and crashes

- **Severity:** 🟠 HIGH · **Status:** ✅ RESOLVED 2026-06-21 · **Category:** api-consistency / ergonomics
- **Resolution:** Library-wide `*,` sweep — only leading column refs stay positional; everything else (`threshold`, `weight`, `pos_label`, `beta`, `alpha`, `delta`, `power`, `eps`, `k`, `log_base`) is now keyword-only across classification, regression, ranking, `roc_auc`, `log_loss`, `brier_score`, `average_precision`, and `gini` (curves/calibration/split/bootstrap were already keyword-only). The `_MetricFn` protocol was updated to match, and the positional-weight footgun now raises a clear `TypeError` at the call site. Benchmark call sites fixed.
- **Location:** `src/polarbearings/classification.py:106-441 (no `*,`); regression.py/roc_auc.py/log_loss.py/brier_score.py/average_precision.py (also positional) vs curves.py:45+, calibration.py:77/266/323, split.py:136, bootstrap.py:110/528 (all keyword-only)`
- **Impact:** Scope is broader than originally framed: the ENTIRE pre-existing codebase (classification + regression + ranking) leaves weight/pos_label/threshold/beta/alpha positional, while every newly-added family forces them keyword-only via `*,`. This produces a real footgun: ranking metrics put weight 3rd-positional (roc_auc('y','p','w') works), but classification metrics put threshold 3rd-positional, so a user with that muscle memory writes f1_score('y','p','w') and gets a cryptic 'ValueError: Unknown format code g for object of type str' from the threshold-token formatter (classification.py:38 _threshold_token f"{threshold:g}" via _alias) — verified for ALL classification metrics (fbeta and non-fbeta alike), not just fbeta as originally claimed. No call-site TypeError; the string silently binds to threshold then crashes downstream during expression construction.
- **Recommendation:** Pick one rule and apply library-wide before 1.0: leading column refs positional; everything else (threshold, weight, pos_label, beta, alpha, delta, n_bins, strategy) keyword-only via `*,`. This is the natural moment (pre-1.0, no stability promise). It simultaneously eliminates the positional-weight footgun (a stray positional weight then raises a clear TypeError at the call site) and the sklearn-vs-here parameter-order mismatches. Apply to ALL families, not just classification.

### 3. Universal null/NaN null-policy contract untested for ~15 newer regression metrics; 100% line-coverage gate structurally cannot catch a forgotten guarded()

- **Severity:** 🟡 MEDIUM · **Status:** ✅ RESOLVED 2026-06-21 · **Category:** test-gap
- **Resolution:** Added `tests/test_null_policy.py` — parametrized over all 39 scalar metrics in `pb.__all__`, asserting `None` on a null target, a NaN value, and (for weighted metrics) a null weight, with a non-null baseline guard against vacuity. A partition test asserts `scalar ∪ non-scalar == set(__all__)`, so a future export must be classified rather than silently skipping the contract the line-coverage gate can't see.
- **Location:** `tests/test_regression.py:1136-1175 (TestMissing asserts only mae, r2_score); tests/test_missing_values.py:18-58; policy routed through _common.py:166 guarded()`
- **Impact:** smape, huber_loss, log_cosh_loss, mean_pinball_loss, mean_squared_log_error, root_mean_squared_log_error, max_error, median_absolute_error, explained_variance_score, the tweedie/poisson/gamma deviance family, and the d2 family are never run through the null harness. Verified all currently DO return None on null/NaN input and null weight — so this is a coverage gap, NOT a live bug. But because every metric routes the policy through the shared guarded() helper and correctness tests execute every line regardless, a future metric that forgets guarded() (or a routing regression for one metric) would ship a non-null result while still hitting the fail_under=100 gate (pyproject.toml:175). Correction: ECE/MCE are NOT part of this gap — test_calibration.py:288-291 already asserts their null policy; the gap is the ~15 regression-family metrics only.
- **Recommendation:** Add a parametrized test iterating the scalar-metric subset of pb.__all__, asserting None on a null target, a NaN pred, and a null weight. This converts the implicit shared-helper assumption into an explicit per-metric guarantee the line-coverage gate cannot provide. Cheap to write; high leverage.

### 4. DCG/NDCG tie handling is row-order-dependent, untested, and diverges from its documented sklearn-parity claim under ties

- **Severity:** 🟡 MEDIUM · **Status:** ✅ RESOLVED 2026-06-21 · **Category:** test-gap / docs-vs-behavior
- **Resolution:** Added `TestTieHandling` in `tests/test_ndcg.py` pinning the order-dependent DCG/NDCG value on tied scores (forward vs reversed input), and qualified the sklearn-parity claim to "for distinct scores" in the ranking module docstring, both `dcg_score`/`ndcg_score` docstrings, and the README — each noting that under ties the result is order-dependent and callers must sort/break ties upstream for reproducibility.
- **Location:** `src/polarbearings/ranking.py:32 (gains_sorted = gain.sort_by(order_by, descending=True)); docstrings lines 11-13,54,85; tests/test_ndcg.py:120-122`
- **Impact:** sort_by preserves input order on equal keys, so physical row order sets the result: df{rel:[3,0],score:[1,1]} -> DCG 3.0 but reversed -> DCG 1.89 (reproduced). Every parity test deliberately uses distinct scores ('Distinct scores avoid tie-ordering ambiguity'), so no test exercises ties at all. Beyond order-dependence, polarbearings actively DIVERGES from sklearn's ignore_ties=True under ties (and breaks the tie the opposite direction: pb 3.0 vs sklearn 1.89), yet the docstrings claim it 'matches/mirrors scikit-learn ... with ignore_ties=True' — a claim only true for distinct scores. For a metric advertised as usable under group_by().agg() and potential streaming, an order-dependent result is a sharp undocumented edge.
- **Recommendation:** Add a test pinning the documented tie behavior on tied scores (lock the order-dependent value for a fixed input order). Reconcile or qualify the sklearn-parity docstring to say parity holds only for distinct scores, and note callers must sort/break ties upstream for reproducibility under grouping. A single-expression design cannot do gain-averaging, so qualify rather than 'fix' the divergence.

### 5. ECE/MCE per-bin filter loop is accidentally quadratic in n_bins (9-68x slower than single-pass group_by)

- **Severity:** 🟡 MEDIUM · **Status:** ✅ RESOLVED 2026-06-21 · **Category:** performance
- **Resolution:** Feature-detected fast path (kept the 1.0.0 floor). `_supports_over_in_agg()` probes for elementwise `Expr.over` inside `group_by().agg()` (pola-rs/polars#25402, added in **polars 1.36.0**, 2025-12-08). On 1.36+ ECE/MCE compute the per-bin means as a single `O(n)` windowed pass that composes in both `select` and `group_by().agg()` (**~4.5x at n_bins=10, ~42x at n_bins=50** on 2M rows); on <1.36 the per-bin `filter` form is the fallback. Both branches are covered by the CI matrix (floor/mid → fallback, latest → fast path); documented in POLARS_COMPATIBILITY.md. No other metric has this per-bin `O(n·k)` pattern — confusion cells are fixed global reductions, and curves/gini/calibration_curve are frame-level single passes — so `over`-in-agg was not applicable elsewhere.
- **Location:** `src/polarbearings/calibration.py:234-250 (per-bin loop, 3 full-column filtered aggs/bin); callers :311, :360`
- **Impact:** On 2M rows / polars 1.0.0: ECE 521ms@n_bins=10 vs 56ms single-pass (9.3x); 15.3s@n_bins=50 vs 224ms (68x). Super-linear in n_bins (~4x per doubling). calibration_curve:151-163 already does the correct single pass. Results agree to ~1e-16 (not literally bit-identical as originally stated; float sum-order).
- **Recommendation:** IMPORTANT constraint the original recommendation missed: ECE/MCE must return a single pl.Expr that composes inside group_by().agg() — a documented, tested guarantee (docstrings 204-206/278-279/335), and the natural .mean().over(binid) replacement is REJECTED on the polars 1.0.0 floor ('window expression not allowed in aggregation'). So do NOT blanket-switch to group_by-on-bin. Instead add a guarded fast-path for the top-level/whole-frame and by cases (where a group_by-on-bin IS expressible), keeping the per-bin filter form only for the nested-in-agg path. Default n_bins=10 already pays 9x.

### 8. Calibration binning + edge-validation logic duplicated across two functions, with a latent quantile divergence if `by` is ever added to ECE/MCE

- **Severity:** 🟡 MEDIUM · **Status:** confirmed · **Category:** duplication / maintainability
- **Location:** `src/polarbearings/calibration.py:40-70 (_bin_edges) vs 191-250 (_calibration_gap_terms)`
- **Impact:** Interior-edge computation, the bin-id loop, and three identical validation strings are implemented twice (error strings duplicated at 51&215, 54&219, 70&227). Quantile is handled differently: _bin_edges resolves edges eagerly and shares them across by-groups (collect().row(0)), while _calibration_gap_terms uses per-group lazy quantile expressions (line 225). They agree today ONLY because ECE/MCE lack a `by` parameter — if `by` were ever added to ECE/MCE, the lazy path would silently compute per-group quantiles, diverging from calibration_curve's shared-edge contract. A correctness landmine for a plausible future feature.
- **Recommendation:** Extract one shared helper for interior edges + bin-id construction plus a shared validator, and decide the quantile-edge semantics once (eager-shared vs per-group) so a future ECE/MCE `by` cannot silently diverge.

- **Impact:** The numpy reference and the implementation compute the identical stable identity (e + log1p(exp(-2e)) - log(2)), so the bulk randomized/property comparison is semi-circular at the formula level — it checks Polars==numpy on the same formula, not formula==definition. Partially mitigated by test_known_value at (0,1) and test_perfect_predictions at e=0, which would catch many wrong-constant errors.
- **Recommendation:** Add one assertion comparing against np.log(np.cosh(pred-y)).mean() on small-residual data (where cosh doesn't overflow), anchoring the stable-identity refactor to the mathematical definition.

### 17. DET / sklearn-roc parity loops filter to shared thresholds with `if t in ours`, which would pass vacuously if thresholds ever shift

- **Severity:** 🟢 LOW · **Status:** confirmed · **Category:** oracle-quality
- **Location:** `tests/test_curves.py:115-121 (DET); tests/test_confusion_curve.py:82-84`
- **Impact:** Both parity loops only assert when the sklearn threshold is present in our output. Currently non-vacuous (1883/1883 DET thresholds match, since both derive from the same scores) — masks nothing today. But the structure depends on exact float-equality of thresholds; a future shift (midpoints/epsilon) would empty the intersection and the test would still pass green. DET fpr has no aggregate oracle, only this filtered point-wise one (DET fnr is backstopped via the ROC AUC property test).
- **Recommendation:** Assert the matched-count > 0 inside the loop so an emptied intersection fails loudly, and/or add a DET monotonicity aggregate (fnr non-increasing, fpr non-decreasing) independent of threshold equality.

### 18. Weighted normalized Gini is NOT 2*weighted-AUC-1 for binary targets — undocumented caveat to the AUC identity

- **Severity:** ⚪ INFO · **Status:** confirmed · **Category:** definition / docs
- **Location:** `src/polarbearings/gini.py:108-146; gini.py:60 docstring; README.md:171,179`
- **Impact:** Unweighted normalized Gini equals 2*AUC-1 exactly for binary targets (verified ~1e-17), but the weighted version does NOT (diffs 0.004-0.375 across trials). The docstring (line 60) and README (171) advertise pos_label -> '2*AUC-1' and README:179 says 'supports optional sample weights' with no caveat that the identity breaks under weighting. Internally consistent (tests pin it to a matching Lorenz reference); no sklearn weighted-Gini analog exists; not a bug. Mechanism is both the per-unit-weight perfect-ordering normalization AND a count-vs-weight Lorenz-axis asymmetry (gini.py:125-126).
- **Recommendation:** Document explicitly that, unlike the unweighted case, weighted binary Gini is not 2*weighted_AUC-1 (it uses a per-unit-weight perfect-ordering baseline), so users don't assume the AUC identity carries to weighted binary data.

### 19. mean_squared_log_error docstring prose implies all negatives NaN; only y < -1 does (and it matches sklearn 1.9 in (-1,0))

- **Severity:** 🟢 LOW · **Status:** partial · **Category:** docs-vs-behavior
- **Location:** `src/polarbearings/regression.py:152-155`
- **Impact:** Down-scoped: the docstring's leading prose ('Negative inputs make the logarithm undefined ... yields NaN for those rows') loosely implies all negatives NaN, while only y < -1 actually NaNs; values in (-1,0) produce a finite result. So the only real issue is the loose docstring prose, not a sklearn divergence — at most low/info.
- **Recommendation:** Tighten the docstring: values < -1 produce NaN; values in (-1,0) produce a finite but out-of-domain result; polarbearings matches sklearn on (-1,0).
