# Performance Analysis

## Methodology

Every metric runs as a native Polars expression in Polars' Rust engine, with no
Python-level loops. Speedup below is **scikit-learn median ÷ polarbearings median**
(higher = polarbearings faster).

Measured on a single machine (Apple Silicon) via this repo's `benchmarks/`,
median/IQR over ≥20 warm rounds with GC disabled (`just bench`). Two Polars
resolutions are compared **ceteris paribus** — numpy and scikit-learn are held
*fixed* and only Polars varies (`just bench-compare`), unlike `--resolution
highest` which would move all three. Absolute times are hardware-specific; the
**ratios** are the portable signal. The tables below are generated by
`benchmarks/compare.py` from the saved runs, which pin **polars 1.41.2** as the newer
leg; the CI test matrix has since moved to 1.42.0 (the benchmark pin lags because
regenerating the numbers needs a controlled, idle machine).

> **On rigor:** a scikit-learn "control" (identical code in both Polars sweeps)
> is tracked to detect machine/thermal drift over long sequential runs. The
> **speedup-vs-scikit-learn** ratios are drift-immune — numerator and
> denominator are measured in the *same* sweep. The **Polars version** ratio is
> not, so it is read drift-corrected (see below).

### Reading these numbers

- **Large, durable wins** on **grouped** metrics and **probabilistic / ranking**
  metrics (Brier, log loss, ROC AUC, precision/F1). These either parallelize
  across groups or do real per-row work, and the advantage holds from small `n`
  all the way to 10M rows.
- **A crossover on trivial reductions** (MAE, MSE, MAPE, R²): faster than
  scikit-learn at small-to-mid `n`, but **slower on a single very large array**,
  where NumPy's tight single-threaded SIMD loop beats one Polars expression. A
  single composable Polars expression runs single-threaded; the parallel win
  returns under `group_by` (which parallelizes across groups) — see below.
- **R² is the slowest** (~0.5× at scale). This was investigated thoroughly: R²
  already uses scikit-learn's exact two-pass mean-deviation formula and compiles
  to a tight 2-node plan. The gap is pure engine overhead, and **no
  pure-expression rewrite beats the current form** while remaining a single
  composable expression (the computational and shifted single-pass formulas are
  slower or numerically unstable; the `pl.struct` form parallelizes but its
  overhead is a net loss at this granularity). Left as-is by design.

Holding numpy/scikit-learn fixed and varying only Polars: newer Polars (1.41)
carries slightly **higher fixed per-call overhead** — most visible as a slowdown
at small `n` — and is roughly **parity at scale** once the run's thermal/ordering
drift is accounted for (the scikit-learn control on the 10M run read ~1.05–1.14,
not 1.0, because the second sweep ran on a warmer machine). The small-`n`
overhead is robust across runs; the at-scale numbers are within drift, so treat
the raw version ratios above as **directional, not exact**. The earlier claim
that "newer Polars widens the gap" did **not** survive clean measurement.

## Why grouped & probabilistic metrics win — and trivial reductions don't, at scale

- **`group_by` parallelizes across groups.** A grouped metric is one vectorized
  pass that Polars spreads across threads, replacing a Python loop that calls
  scikit-learn once per group — the largest speedups in the suite.
- **A single composable expression is single-threaded.** Polars parallelizes
  across *independent output columns* and *groups*, not across the sub-parts of
  one fused scalar expression. For a whole-frame single scalar (e.g. one MAE over
  10M rows) Polars runs on one thread, and NumPy's tighter loop wins.
- **Trivial reductions are memory-bandwidth-bound.** Summing 10M f64 ≈ 80 MB;
  one SIMD thread nearly saturates memory bandwidth, so threads can't help and
  per-element Arrow overhead makes Polars ~2× slower than NumPy on the cheapest
  metrics at scale. This is structural, not a missing optimization.

## Performance tips

1. **Use grouped operations** — calculate metrics per group in one pass (where
   the largest wins are):
   ```python
   df.group_by("group").agg(roc_auc("label", "score"))
   ```
2. **Batch metrics in one `select`** — share the scan across several metrics:
   ```python
   df.select(
       roc_auc("label", "score"),
       log_loss("label", "prob"),
       brier_score("label", "prob"),
   )
   ```
3. **Filter first** to reduce work, and **stay lazy** for complex pipelines:
   ```python
   (df.lazy()
      .filter(pl.col("label").is_not_null())
      .group_by("segment")
      .agg(roc_auc("label", "score"))
      .collect())
   ```
4. **For a one-off metric over a single giant array**, NumPy/scikit-learn is a
   fine choice — that's the one shape where a single Polars expression doesn't win.
