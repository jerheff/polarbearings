## New findings (prioritized)

### 1. `hash_split`/`hash_fold`/`hash_uniform` reproducibility silently depends on Polars' hash, which is NOT stable across Polars versions — undocumented, and no test would catch a reshuffle

- **Severity:** 🟠 HIGH · **Category:** correctness-of-promise / docs / test-gap
- **Location:** [split.py:45–66](../src/polarbearings/split.py) (`hash_uniform` → `Expr.hash`); consumed by `hash_split`, `hash_fold`, `hash_splits`, and `bootstrap_weight` (bootstrap.py:201–216)
- **Evidence:** Polars' own `Expr.hash` docstring (verified on the installed 1.0.0): *"This implementation of `hash` does not guarantee stable results across different Polars versions. Its stability is only guaranteed within a single version."*
- **Impact:** The module's headline promise is durable, id-keyed splits — "reproducible across runs and row orderings," "a holdout stays stable as the dataset or the fraction changes," "the same id always lands in the same split, even across datasets." All of that is true **only within one Polars version**. A user who pins their holdout with `hash_split(seed, "id", fraction=0.2)`, trains for months, then upgrades Polars can have *every* id silently reassigned — the classic train/test contamination failure the module exists to prevent. Nothing in split.py, README, or METRICS.md mentions this. And `tests/test_split.py` asserts only *properties* (determinism within-process, subset-consistency, rate), never pinned hash values, so a cross-version reshuffle would sail through the entire CI matrix green. The same instability makes `bootstrap_weight` draws version-dependent, but that only perturbs CI endpoints statistically — far less severe.
- **Recommendation (layered):**
  1. **Document now** (cheap, honest): a prominent caveat in the split.py module docstring, both README/METRICS split sections, and `bootstrap_weight`'s docstring — splits are stable *per Polars version*; for a long-lived holdout, materialize the assignment (or the `hash_uniform` value) to storage once and join it back, rather than recomputing.
  2. **Add a canary test**: pin ~5 known `(id, seed) → uniform` values for the current version, marked so the multi-version CI legs surface the exact Polars release that changes the hash — turning a silent reshuffle into a loud, versioned fact you can document.
  3. **Longer term**, consider a version-independent hash for the split path (e.g. a splitmix64-style integer mix built from wrapping arithmetic on a `UInt64` cast of the id, or documenting a recipe where the user supplies a precomputed stable key). That trades a little speed for an actually-durable promise; worth a design note even if deferred.

### 2. `bootstrap()`'s non-weightable guard is incomplete: `dcg_score`, `ndcg_score`, `d2_absolute_error_score`, `d2_pinball_score` raise a raw `TypeError` instead of the intended friendly `ValueError`

- **Severity:** 🟡 MEDIUM · **Category:** error-quality / maintainability
- **Location:** [bootstrap.py:64](../src/polarbearings/bootstrap.py) (`_NON_WEIGHTABLE = {"max_error", "median_absolute_error"}`), used at bootstrap.py:296
- **Evidence (repro):** `bootstrap(dcg_score, "y", "p")` → `TypeError: dcg_score() got an unexpected keyword argument 'weight'`; same for the other three. `max_error` correctly gets `ValueError: bootstrap() needs a weightable metric...`.
- **Impact:** Six exported metrics lack `weight`, but the hand-maintained frozenset only lists two — it went stale the moment the d2/ranking metrics landed and will go stale again. Users hitting the `TypeError` get no pointer to the actual constraint (weighted-bootstrap requires a weightable metric).
- **Recommendation:** Replace the hardcoded set with a signature check — `"weight" not in inspect.signature(metric).parameters` (unwrapping `functools.partial` the way `_metric_name` already does) — so the friendly error can never drift from reality. One test parametrized over the unweightable exports locks it.

### 3. `bootstrap_ci` on degenerate data dies with an opaque `TypeError` instead of reporting the metric is undefined

- **Severity:** 🟡 MEDIUM · **Category:** error-quality / UX
- **Location:** [bootstrap.py:520–522](../src/polarbearings/bootstrap.py) (`float(materialized...item())`, `_reduce_ci` sorting the distribution)
- **Evidence (repro):** `bootstrap_ci(df_single_class, roc_auc, "y", "p")` → `TypeError: float() argument must be a string or a real number, not 'NoneType'`.
- **Impact:** Every metric in the library deliberately returns null for undefined cases (single-class AUC, constant-target R², …). Bayesian bootstrap weights are strictly positive, so if the point estimate is null, *all* replicates are null too — `bootstrap_ci` is guaranteed to crash on exactly the inputs where the library's null convention was supposed to degrade gracefully. The per-group `by=` path has the same hazard for any single group (nulls flow into `ci_from_distribution`'s list ops, silently yielding null bounds — inconsistent with the whole-frame crash).
- **Recommendation:** Detect a null estimate before reducing and return `{"estimate": None, "low": None, "high": None}` (mirroring the expression-level convention), or raise a `ValueError` explaining the metric is undefined on this data. Pick one and add a degenerate-input test for both the whole-frame and `by=` paths.

### 4. All-zero (or zero-sum) weights yield `NaN` from mean-style metrics but `null` from count-style metrics — an inconsistent edge policy

- **Severity:** 🟢 LOW · **Category:** consistency
- **Location:** every `(x * w).sum() / w.sum()` site — e.g. [log_loss.py:50](../src/polarbearings/log_loss.py), [brier_score.py:43](../src/polarbearings/brier_score.py), regression.py throughout — vs. the `denom == 0 → null` guards in classification.py and `single_class → null` in roc_auc.py
- **Evidence (repro):** with `w = [0.0, 0.0, 0.0]`: `mae` → `NaN`, `log_loss` → `NaN`, but `precision` → `null`, `roc_auc` → `null`.
- **Impact:** The library's documented convention is "null for undefined cases," and a zero-total-weight aggregate is undefined. Half the metrics follow the convention; the other half leak a `0/0 = NaN`. Pathological input, but the split behavior makes the null-policy contract (which `tests/test_null_policy.py` otherwise locks) look accidental at this edge, and NaN vs null behave differently downstream (`is_null`, joins, `fill_null`).
- **Recommendation:** Either wrap the weighted-mean denominators (`pl.when(w.sum() > 0)...otherwise(None)` — cheap, one shared helper in `_common.py` would DRY the eleven duplicated `(x*w).sum()/w.sum()` sites at the same time), or explicitly document NaN-on-zero-weight as the accepted behavior. Add the zero-weight case to the null-policy test either way.

### 5. Calibration binning still split across two parallel implementations, and the quantile-edge semantics of ECE/MCE vs `calibration_curve` differ without being called out

- **Severity:** 🟢 LOW · **Category:** duplication / docs (residual of June finding 8)
- **Location:** [calibration.py:62–92](../src/polarbearings/calibration.py) (`_bin_edges`, eager whole-frame edges) vs [calibration.py:213–253](../src/polarbearings/calibration.py) (`_bin_setup`, lazy per-context quantile exprs); inline bin-id loop duplicated at 164–166 and 250–252; validation strings duplicated at 73/235, 76/239, 92/247
- **Impact:** Beyond the maintenance duplication, there is a real (currently latent) semantic fork: `calibration_curve(..., strategy="quantile", by=...)` deliberately shares whole-frame edges across groups (documented at calibration.py:127–130), while `expected_calibration_error(..., strategy="quantile")` used inside `group_by().agg()` computes **per-group** quantile edges (the lazy expressions in `_bin_setup` resolve within each group). Both choices are defensible; the fact that they differ is documented nowhere — a user comparing a per-group ECE against a `by=` calibration curve on the same segments is silently using different bins.
- **Recommendation:** Consolidate edges/bin-id/validation into one helper with an explicit `edge_scope` choice, and add one sentence to the ECE/MCE docstrings: "under `group_by().agg()`, `strategy='quantile'` bins each group at its own quantiles (unlike `calibration_curve`'s shared edges)."

### 6. `dcg_score`/`ndcg_score` accept nonsensical `k` values: `k=-1` fails at collect time with an opaque Polars `ComputeError`; `k=0` is meaningless

- **Severity:** 🟢 LOW · **Category:** input-validation / error-quality
- **Location:** [ranking.py:38–40](../src/polarbearings/ranking.py) (`head(k)` with unvalidated `k`)
- **Evidence (repro):** `dcg_score("rel", "score", k=-1)` collects to `ComputeError: unable to extract length from shape: (1,) ... slice(offset=0, length=-1)` — deep engine internals, no hint the user's `k` is the cause (Polars' `head(-n)` means "all but the last n", so other negative values may *silently compute the wrong thing* instead of erroring).
- **Recommendation:** Validate eagerly in Python: `if k is not None and k < 1: raise ValueError(...)`. Two lines, converts a confusing engine error (and a potential silent wrong answer) into an immediate, named failure.

### 7. `balanced_sample_weight` silently treats null labels as their own class, shifting every weight; no null policy is documented

- **Severity:** 🟢 LOW · **Category:** null-policy consistency
- **Location:** [class_weight.py:43–48](../src/polarbearings/class_weight.py) (`pl.len().over(target)`, `n_unique()` — both count null as a class); `balanced_class_weights` (66–73) has the same behavior via `value_counts`
- **Evidence (repro):** `y = [1, 1, 0, None]` → `n_classes` becomes 3, the null row gets weight 1.33, and the real classes' weights shift (class 1 → 0.67 instead of the 2-class 1.0/1.5). scikit-learn's `compute_sample_weight` would not quietly do this.
- **Impact:** Every metric in the library has an explicit, tested missing-input policy (`guarded()` → null, or documented row-dropping for curves); the class-weight helpers are the only public functions with *no* stated policy, and their actual behavior — null becomes a phantom class that dilutes all other weights — is the least defensible option.
- **Recommendation:** Pick a policy and document it. Most consistent with the library: null out the result-row weight for null labels and exclude them from `n_classes`/counts (`pl.when(target.is_null()).then(None)` plus `drop_nulls`-style counting), matching the "missing input never silently changes a number" principle.

### 8. Smaller consistency & hygiene items

- **`pos_label` typed inline instead of the shared alias** — `average_precision`, `log_loss`, and `brier_score` spell `pos_label: int | float | str | bool` while `roc_auc`/`gini`/classification use `PosLabel` ([average_precision.py:21](../src/polarbearings/average_precision.py), log_loss.py:22, brier_score.py:21). Cosmetic, but the alias exists precisely so the family can't drift.
- **`weight_suffix` collapses every expression weight to `"_w"`** ([_common.py:120](../src/polarbearings/_common.py)) — two calls to the same metric with two *different* weight expressions in one `select` collide on the alias and raise a duplicate-column error. Rare; a short hash of `output_name()` or a documented "alias yourself when passing two weight exprs" note would cover it.
- **`any_missing()` raises `IndexError` if called with no columns** ([_common.py:174](../src/polarbearings/_common.py) `flags[0]`) — internal-only today; a guard or a `pl.lit(False).any()` seed would make the helper total.
- **`percentile_thresholds` is a legacy odd-one-out** ([classification.py:523](../src/polarbearings/classification.py)) — the only eager, `Series`-in/`list[float]`-out function in the metric API, fully subsumed by the lazy `quantiles(n)` spec (which also works per-group). Candidate for deprecation before 0.1.0 locks the API surface.
- **`_GRID_EXACT_CUTOVER=30` is self-admittedly provisional** ([confusion_curve.py:36–43](../src/polarbearings/confusion_curve.py) — tuned on a thermally-throttling laptop per its own comment). Re-tune on a stable box, or leave the honest comment and revisit post-release.
- **`just ci` ≠ CI** — `ci: quality test` omits the coverage gate, the compat matrix, and `test-highest` that the workflows enforce, while CLAUDE.md sells it as "Run all CI checks locally." Either extend the recipe (`ci: quality test test-cov`) or re-word the description.
- **Stale local `mutants/` tree uses the pre-rename package name `polarbear`** — untracked and `just clean` removes it, but at ~25k lines it dominates naive greps/searches (it inflated this audit's first file survey too). Worth a `rm -rf mutants/` when not actively mutation-testing.
- **Latest-Polars pin is hardcoded in 5+ places** (justfile `test-compat`/`test-cov`/`bench-compare`, test.yml matrix and coverage legs, docs) — the recurring "bump recent Polars compat slot" commits do this by hand each release. A single source (e.g. a `latest_polars := "1.42.1"` justfile var + one workflow env) would shrink the bump to two lines. Minor, recurring toil.

## Positive observations

- **The numerical core held up to a fresh full read.** Overflow-conscious rank AUC (UInt64 cast with the reason documented), variance-free tie detection (`max == min` to dodge underflow), stable log-cosh identity, sklearn-exact `inverted_cdf` pinball baseline via `implode().list.get` with the 1.24.0 panic workaround explained — every nontrivial choice carries a comment explaining *why*, which is rarer than it should be.
- **June-audit remediation was real, not cosmetic**: keyword-only sweep, null-policy contract test parametrized over `__all__` with a partition guard, branch-coverage-via-cross-version-combine replacing all pragmas, tie-handling docs that honestly state where sklearn parity ends.
- **The missing-data design is coherent and legible**: scalar metrics null the context, curves drop incomplete rows, both documented at every entry point, with the value/label column distinction (`NaN` checked only where it can exist) both correct and performance-justified.
- **Engineering infrastructure punches above its weight**: floor-resolution dev env, feature-detected (never version-parsed) fast paths with both arcs CI-covered, deprecations escalated to errors on the highest leg, wheel tested as-installed, honest benchmark methodology (thermal cooldown, ceteris-paribus version comparisons), and 679 tests including property-based and multi-version legs.
